# 本体模型设计器 · 使用手册

> 适用版本：Ontology Designer MVP（Vue 3 + TypeScript + Vite）。
> 本手册帮助你**从零为任意业务搭建本体模型**。建模理论见 [本体模型设计规格书.md](本体模型设计规格书.md)，内置范例解读见 [合同管理系统-场景与需求.md](合同管理系统-场景与需求.md)。

---

## 1. 快速开始

### 1.1 启动

```powershell
cd designer
npm install      # 首次运行
npm run dev
```

浏览器打开终端提示的地址（默认 `http://localhost:5173/`，若端口被占用会自动顺延，如 `5174`）。

### 1.2 构建产物

```powershell
cd designer
npm run build    # 输出到 designer/dist，可静态部署
npm run preview  # 本地预览构建结果
```

### 1.3 第一次上手

打开后顶栏点击 **「加载示例」**，即载入"合同管理系统"完整本体，可作为学习样板；点击 **「新建空白」** 则清空，开始你自己的建模。

## 2. 界面总览

整体为「顶栏 + 三栏」布局：

```
┌─────────────────────────────────────────────────────────────┐
│ 顶栏：设计器标题 │ 项目名 │ 校验状态 │ [加载示例] [新建空白] │
├──────────┬───────────────────────┬──────────────────────────┤
│ 左栏      │ 中栏                  │ 右栏                      │
│ 模型导航  │ 模型编辑器             │ 可视化 / 校验 / YAML       │
│ + 列表    │ （表单）               │ （4 个标签页）             │
└──────────┴───────────────────────┴──────────────────────────┘
```

| 区域 | 作用 |
|---|---|
| **顶栏** | 改项目名；实时看校验结果（`✓ 校验通过` / `✖ N 错误` / `N 警告`）；加载示例 / 新建空白 |
| **左栏** | 四类模型切换（OBJ/BHV/EVT/RULE，带数量徽标）+ 当前类型的模型列表（新建 / 选中 / 删除）|
| **中栏** | 选中模型的表单编辑器，随类型自动切换 |
| **右栏** | 事件链图 / 聚合图 / 校验明细 / YAML 导入导出 |

### 2.1 四类核心模型

| 代号 | 名称 | 回答 | 颜色 | 直觉 |
|---|---|---|---|---|
| **OBJ** | 对象模型 | 是什么 | 蓝 | 业务实体/聚合（名词）|
| **BHV** | 行为模型 | 做什么 | 绿 | 用例/操作（动词）|
| **EVT** | 事件模型 | 如何传播 | 橙 | 已发生的事实（过去式）|
| **RULE** | 规则模型 | 为什么 | 紫 | 约束与决策逻辑 |

## 3. 编辑各类模型

左栏选中某类型后点 **「+ 新建」** 创建一条，再在列表中点击它，中栏即出现对应表单。**所有改动实时生效并自动保存**（无需手动保存按钮）。

### 3.1 OBJ 对象模型

| 区块 | 字段 | 说明 |
|---|---|---|
| 基本 | id / 名称 / 唯一标识字段 / 描述 | `唯一标识字段`填聚合根的主键属性名（如 `orderNo`）|
| **属性** | 属性名 / 类型 / 必填 / 说明 | 类型见下表；每行可增删 |
| **子实体** | id / 名称 / 标识字段 + 属性 | 聚合内、依赖聚合根而存在（如订单明细行）|
| **值对象** | id / 名称 + 属性 | 无独立标识、不可变（如地址、金额）|
| **不变性约束** | id / 表达式 / 说明 | 聚合内恒成立的业务规则（如 `amount > 0`）|
| **对外引用** | 目标聚合 / 引用字段 / 说明 | 以 ID 引用另一聚合（如 `customerId → Customer`）|

属性类型（`类型`下拉）：

| 类型 | 含义 | 类型 | 含义 |
|---|---|---|---|
| `string` | 文本 | `datetime` | 日期时间 |
| `number` | 整数/浮点 | `enum` | 枚举 |
| `decimal` | 精确小数（金额）| `reference` | 引用其他对象 |
| `boolean` | 布尔 | `object` | 嵌套对象 |
| `date` | 日期 | | |

> **建模要点**：一个 OBJ 即一个**聚合**，以聚合根为边界。跨聚合只能用「对外引用」持有对方 ID，不要直接嵌套对方实体。

### 3.2 BHV 行为模型

| 区块 | 说明 |
|---|---|
| id / 名称 / **所属聚合根** / 描述 | `所属聚合根`下拉选一个 OBJ，表示该行为操作哪个聚合 |
| 前置条件 | 执行前需满足的状态（多行，可增删）|
| 后置条件 | 执行后保证的状态 |
| **应用的规则** | 多选 RULE：行为执行时同步调用（如计算、风控）|
| **产生的事件** | 多选 EVT：行为成功后发布的事件 |
| **订阅的事件** | 多选 EVT：监听到该事件即触发本行为 |

> **编排核心**：行为之间不直接调用，而是通过「产生事件 → 另一行为订阅事件」解耦串联，形成事件链。

### 3.3 EVT 事件模型

| 区块 | 说明 |
|---|---|
| id / 名称 / **顺序语义** / 描述 | 命名建议用过去式（`OrderPaid`）|
| 事件载荷 payload | 事件携带的数据字段（同属性编辑器）|

顺序语义（投递保证）：

| 值 | 含义 |
|---|---|
| `AT_LEAST_ONCE` | 至少一次（消费端需幂等）— 默认推荐 |
| `EXACTLY_ONCE` | 恰好一次（强一致，成本高）|
| `BEST_EFFORT` | 尽力而为（可丢失，如通知）|

### 3.4 RULE 规则模型

| 区块 | 说明 |
|---|---|
| id / 名称 / **规则类型** / 描述 | 类型见下 |
| 条件判断 | 规则的判定逻辑（自然语言或表达式）|
| 订阅的事件 ＊ | 仅「事件驱动规则」可见：触发本规则的事件 |
| 触发的事件 ＊ | 仅「事件驱动规则」可见：满足条件后发布的事件 |

规则类型：

| 类型 | 用途 | 参与事件链 |
|---|---|---|
| `validation` 验证 | 数据/状态校验 | 否 |
| `calculation` 计算 | 派生计算（金额、税）| 否 |
| `derivation` 推导 | 推导赋值 | 否 |
| `risk` 风控 | 风险判断 | 否 |
| `event-driven` 事件驱动 | **智能决策节点** | ✅ 订阅事件→判断→触发事件 |

> 前 4 类是被行为「应用的规则」同步调用；只有 `event-driven` 能成为事件链上的独立决策节点（订阅→触发）。

## 4. 右栏：可视化与校验

### 4.1 事件链（默认）

把 BHV / EVT / RULE 按「产生 / 订阅 / 触发」关系渲染成有向图，直观呈现整条业务流与闭环。可缩放、适配视图。**这是检查编排是否连通、是否有断点的主视图。**

### 4.2 聚合图

选一个 OBJ，展示其聚合根、子实体、值对象与对外引用的结构关系。用于核对聚合边界设计。

### 4.3 校验

实时列出所有一致性问题。点击条目可定位。校验规则：

| 代码 | 级别 | 含义 |
|---|---|---|
| `DUPLICATE_ID` | 错误 | 同类型存在重复 id |
| `DANGLING_OBJ_REF` | 错误 | 对外引用指向不存在的聚合 |
| `DANGLING_OBJECT` | 错误 | 行为归属的聚合不存在 |
| `DANGLING_RULE` | 错误 | 引用了不存在的规则 |
| `DANGLING_EVENT` | 错误 | 引用了不存在的事件 |
| `EVENT_CYCLE` | 错误 | 事件-规则图存在环路 |
| `EVENT_RULE_NO_SUB` | 警告 | 事件驱动规则未订阅任何事件 |
| `ORPHAN_EVENT` | 警告 | 事件既无生产者也无订阅者 |

目标是顶栏显示 **`✓ 校验通过`**（0 错误 0 警告）。

### 4.4 YAML

- **导出**：将整个本体序列化为 YAML，可复制保存到版本库，作为 AI 大模型/代码生成的输入契约。
- **导入**：粘贴 YAML 后点「解析并导入」，经 Schema 校验通过即替换当前项目（失败会提示错误，不会破坏现有数据）。

---

## 5. 数据持久化

- 所有改动**自动保存**到浏览器 IndexedDB（防抖约 400ms），刷新/重开自动恢复上次内容。
- **「加载示例」** 会用内置范例**覆盖**当前内容；**「新建空白」** 会清空当前内容——请先用 YAML 导出备份重要数据。
- 想长期保存或多人协作：用 **YAML 导出**，存入 Git 等版本库。

## 6. 实战：从零搭建一个新业务本体（请假审批）

下面以"员工请假审批"演示完整建模流程，套路可复制到任意业务。

### 步骤 0 · 准备

顶栏点「新建空白」，把项目名改为 `请假审批系统`。

### 步骤 1 · 先建 OBJ（是什么）

切到 **OBJ**，「+ 新建」两个聚合：

1. `LeaveRequest` 请假单（唯一标识字段 `requestId`）
   - 属性：`requestId`(string,必填)、`employeeId`(reference)、`type`(enum：年假/病假/事假)、`days`(decimal)、`reason`(string)、`status`(enum：草稿/待审批/已批准/已驳回)
   - 不变性约束：`inv_days` 表达式 `days > 0` 说明"请假天数为正"
   - 对外引用：目标聚合 `Employee`、引用字段 `employeeId`
2. `Employee` 员工（唯一标识字段 `employeeId`）
   - 属性：`employeeId`(string,必填)、`name`(string)、`annualQuota`(decimal)、`annualUsed`(decimal)
   - 不变性约束：`inv_quota` 表达式 `annualUsed <= annualQuota`

### 步骤 2 · 再建 EVT（如何传播）

切到 **EVT**，新建事件（顺序语义默认 `AT_LEAST_ONCE`）：

- `LeaveSubmitted` 请假已提交
- `LeaveApproved` 请假已批准
- `LeaveRejected` 请假已驳回
- `QuotaInsufficient` 额度不足

### 步骤 3 · 建 RULE（为什么）

切到 **RULE**，新建一个**事件驱动规则**：

- `QuotaCheckRule` 额度检查（类型选 `event-driven`）
  - 条件判断：`年假类型且 annualUsed + days <= annualQuota`
  - 订阅的事件：`LeaveSubmitted`
  - 触发的事件：`LeaveApproved`（满足）——演示用，真实可拆双分支

### 步骤 4 · 建 BHV（做什么）并接线

切到 **BHV**，新建行为，逐一设置「所属聚合根 / 订阅事件 / 产生事件 / 应用规则」：

| 行为 | 所属聚合 | 订阅 | 产生 |
|---|---|---|---|
| `SubmitLeave` 提交请假 | LeaveRequest | — | LeaveSubmitted |
| `ApproveLeave` 批准请假 | LeaveRequest | LeaveApproved | — |

这样就形成闭环：**提交请假 →（LeaveSubmitted）→ 额度检查规则 →（LeaveApproved）→ 批准请假**。

### 步骤 5 · 验证与导出

1. 看右栏 **事件链**：确认上述节点连成一条通路、无断点。
2. 看顶栏校验：应为 **`✓ 校验通过`**；若有红色错误，切到 **校验** 标签按提示修正（常见：事件名拼写不一致导致悬空引用）。
3. 切到 **YAML** → 导出 → 复制保存到你的版本库。

完成。把"请假"换成你的真实业务名词、动词、事件、规则，即可建出任意领域的本体。

## 7. 建模建议（推荐顺序与心法）

1. **名词 → OBJ**：先圈出业务实体与聚合边界（一个聚合一致务边界）。
2. **事实 → EVT**：列出"已发生的事实"，用过去式命名。
3. **决策 → RULE**：把"判断/计算/风控"沉淀为规则；需要在流程中自动决策的用 `event-driven`。
4. **动作 → BHV**：定义操作，用「订阅/产生事件」把行为串成事件链，用「应用规则」挂同步规则。
5. **校验直到 `✓`**：以"事件链连通 + 0 错误 0 警告"为完成标准。
6. **导出 YAML 固化**：作为与 AI 大模型/代码生成协作的契约。

### 常见问题

| 现象 | 原因 / 处理 |
|---|---|
| 改了示例代码但页面没变 | 浏览器加载的是 IndexedDB 持久化数据；点「加载示例」刷新 |
| 顶栏报"悬空引用"错误 | 行为/规则引用的事件或聚合 id 不存在或拼错，按校验提示修正 |
| 报"检测到环路" | 事件-规则形成闭环回路，调整订阅/触发关系打破环 |
| 事件显示"孤立事件"警告 | 该事件既没行为产生也没被订阅，补生产者或消费者，或删除 |
| 事件驱动规则报"未订阅事件" | 给该规则补「订阅的事件」 |
| 误点"新建空白"丢数据 | 数据已被覆盖；下次操作前先 YAML 导出备份 |
