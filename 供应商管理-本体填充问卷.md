## 🎤 R0 · 立项与范围

> **目标**：圈定系统边界、关键角色、登记原始需求（建立"不漏"的标尺）。

| # | 顾问问法 | 提示 / 追问 | 参考示例（合同管理，→落到字段） | ✍️ 我的回答 |
|:--:|---|---|---|---|
| 1 | 这套供应商管理系统先管哪一块？用一两句话说它要解决什么。 | 管到哪为止？哪些**不**管？（采购下单？付款结算？还是只管供应商档案、准入、资质？） | "管理 B2B 销售合同从草拟到履约到期的全流程"（→ `name`/`description`） |  |
| 2 | 平时**谁**会用它？各自干什么？ | 谁录入/申请、谁审批、谁评估打分？有无外部系统（ERP/采购/征信）？ | `SalesRep` 销售代表、`SalesManager` 销售经理（→ `actors[]`） |  |

## 🎤 R1 · 对象与关系

> **目标**：识别核心业务对象、属性、聚合边界与对象间关系。

| # | 顾问问法 | 提示 / 追问 | 参考示例（合同管理，→落到字段） | ✍️ 我的回答 |
|:--:|---|---|---|---|
| 1 | 这摊业务里有哪些"东西/单据"？哪个是主角？ | 哪个是核心单据？哪些是挂在它下面的明细？ | `Contract`（合同）、`Customer`（客户）、`Delivery`（配送单）（→ `objects[]`） |  |
| 2 | 每一类，靠什么**编号**区分彼此？ | 编号有规则吗？ | `Contract.contractNo`（合同号）（→ `identity`） |  |
| 3 | 每一类上要记哪些**信息**？ | 有金额吗（精确到分）？有日期吗？ | 合同号/金额(decimal)/状态(enum)/到期日(date)（→ `attributes[]`） |  |
| 4 | 一份主单能装多条**明细**吗？ | 这些明细能脱离主单单独存在吗？ | `ContractItem` 子实体；`ShippingAddress` 值对象（→ `entities[]`/`valueObjects[]`） |  |
| 5 | 有没有"**任何时候都必须对得上**"的铁规矩？ | 比如总额必须等于各明细之和？ | `amount == sum(items.subtotal)`；明细数 ≥ 1（→ `invariants[]`） |  |
| 6 | 各对象之间怎么**关联**？数量上几对几？ | 一个 A 能对几个 B？一个 B 属于几个 A？ | `Contract→Customer` 多对一（→ `references[]`+`cardinality`） |  |
| 7 | 删掉一个主对象时，它的明细、它关联的对象分别会怎样？ | 哪些随它一起删、哪些要保留？ | Customer 为 `association`（删合同不删客户）；ContractItem 为 `composition`（随合同删）（→ `references[].kind`） |  |

## 🎤 R2 · 状态与生命周期

> **目标**：为有状态流转的对象建状态机（状态 + 迁移，迁移先占位动作名）。

| # | 顾问问法 | 提示 / 追问 | 参考示例（合同管理，→落到字段） | ✍️ 我的回答 |
|:--:|---|---|---|---|
| 1 | 哪个对象有"从生到死的状态流转"？会经历哪些**状态**？ | 从哪个状态开始？哪些状态算"走到头了"？ | 草稿→审批中→生效→（到期/终止）（→ `lifecycle.states[]`） |  |
| 2 | 用哪个字段记录它**现在到哪一步**了？ | 取值和上面列的状态对得上吗？ | `status`（→ `lifecycle.stateAttr`） |  |
| 3 | 从一个状态走到下一个，靠什么**动作**推动？ | 有没有前提条件？谁来推动？ | 提交/激活/驳回/到期/终止（→ `transitions[].onBehaviorRef`，占位） |  |
| 4 | 有没有"走不到头"或"回不去"的状态？ | 比如被驳回后还能重新提交吗？ | 审批中可驳回退回草稿（→ 迁移完整性） |  |

## 🎤 R3 · 行为与触发

> **目标**：定义对象上的原子操作，补全 R2 占位迁移动作，明确触发方式与前后置。

| # | 顾问问法 | 提示 / 追问 | 参考示例（合同管理，→落到字段） | ✍️ 我的回答 |
|:--:|---|---|---|---|
| 1 | 围绕这些单据，大家平时会做哪些**操作**？每个动哪类单据？ | 尽量列全（增删改、状态推进、生成下游单据等） | 提交审批/激活/驳回/到期失效/人工终止/建开票计划/建配送单/发到期提醒（→ `behaviors[].objectRef`） |  |
| 2 | 每个操作怎么**发起**？ | 人工点的？被前一件事带出来的？还是到点自动跑？ | 提交=manual；建配送=event；到期失效=timer（→ `trigger.kind`） |  |
| 3 | 人工的话**谁能点**；自动的话跟着哪件事、或什么时间跑？ | 每天定点，还是到某个日子才跑？ | 激活=SalesManager；到期提醒=每日 9 点且到期前 7 天（→ `trigger.actorRef/eventRefs/schedule/deadline`） |  |
| 4 | 做这操作**之前**必须满足什么？做完**之后**系统保证什么？ | 条件不满足会怎样？ | 前置"审批通过且首付到账"→后置"状态=生效"（→ `pre/postconditions`） |  |
| 5 | 这操作万一做错了，怎么**补救**？ | 有没有一个反向操作能撤回它？ | 激活 ↔ 终止（→ `compensatedBy`） |  |

## 🎤 R4 · 规则与策略

> **目标**：抽取可复用纯规则（RULE）与事件驱动反应（POLICY）。

| # | 顾问问法 | 提示 / 追问 | 参考示例（合同管理，→落到字段） | ✍️ 我的回答 |
|:--:|---|---|---|---|
| 1 | 有哪些**判断或计算**，是反反复复要用到的？ | 它是"检查对不对"、"算个数额"、还是"卡风险"？ | `CreditLimitCheck`（校验型）：已用额度+本单 ≤ 授信（→ `rules[].type`+`condition`） |  |
| 2 | 这条判断**只在某一步**用，还是到处都用？ | 只在某操作时查，还是多处都查？ | 信用校验在激活时被引用（→ RULE vs 行为前后置） |  |
| 3 | 有没有"**一发生某件事，就自动做另一件事**"的规则？ | 哪件事触发它？它做完又引出哪件事？ | `DeliveryConditionCheck`：收到"合同激活"→（含实物且地址确认）→请求配送（→ `policies[].subscribed/triggeredEventRefs`） |  |

## 🎤 R5 · 事件闭环

> **目标**：定义事件载荷与投递语义，连通"行为→事件→策略→事件→行为"，做环路检测。

| # | 顾问问法 | 提示 / 追问 | 参考示例（合同管理，→落到字段） | ✍️ 我的回答 |
|:--:|---|---|---|---|
| 1 | 前面这些"一发生就要通知别人"的事，每条通知里要带上哪些**信息**？ | 收到通知的人，得有哪些信息才能接着干活？ | `ContractActivated{contractNo,productType}`（→ `events[].payload`） |  |
| 2 | 这条通知可以**偶尔丢**吗？要不要保证只处理一次？ | 万一同一条发重了，接收方受得了吗？ | 多数 `AT_LEAST_ONCE`；提醒 `BEST_EFFORT`（→ `deliverySemantics`） |  |
| 3 | 每条通知都有"谁发出"和"谁接收"吗？ | 有没有发了没人理、或没人发却有人等的？ | 5 事件均有产销（→ 生产/订阅连通） |  |
| 4 | 顺着这些通知一路传下去，会不会**绕回自己**？ | 比如 A 引出 B、B 又反过来引出 A？ | 链路无环（→ 事件图无环，`EVENT_CYCLE=0`） |  |

## 🎤 R7 · 治理与追溯

> **目标**：统一术语、把每个核心元素溯源到需求、补关键授权与性能要求。

| # | 顾问问法 | 提示 / 追问 | 参考示例（合同管理，→落到字段） | ✍️ 我的回答 |
|:--:|---|---|---|---|
| 1 | 业务里那些关键"**黑话**"，各自到底什么意思？有别名吗？ | "准入""黑名单""资质"具体指什么？ | `生效`=签字且首付到账后；别名"激活"（→ `glossary[]`） |  |
| 2 | 这个词在系统里对应哪份**单据/操作/状态**？ | 它靠哪个操作实现？ | 生效 ↔ `Contract/ActivateContract/ContractActivated`（→ `relatedElementRefs`） |  |
| 3 | 我们建的每样东西，分别是为满足你提的**哪条要求**？ | 找不到对应要求的，是不是多余的？ | 自动配送行为 ← `req_auto_delivery`（→ 元素 `requirementRefs`） |  |
| 4 | 你提的那些要求，是不是**每一条都落实**了？ | 有没有"提了却没做"的？ | 3 条确认需求全被引用（→ 双向追溯，必须无孤儿需求） |  |
| 5 | 对外的关键操作，**谁有权**做？ | 没权限的人去做会怎样？ | 激活仅 `SalesManager`（→ `BHV.authorize`） |  |
| 6 | 关键操作有**快慢或可靠性**要求吗？ | 多快算达标？要多高可用？ | 快写 `qos_fast_write` 500ms/0.999（→ `qosProfiles`+`BHV.qos`） |  |
